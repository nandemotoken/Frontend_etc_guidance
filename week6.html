第6回:フロントエンドの壊れかた

― 全員が犯人、誰も悪くない（？）―

専門性が生んだ悲劇であり、喜劇

今日の物語

鳴り物入りでリリースされた新フロントエンド「The・World」。

しかし、その直後、画面は真っ白に。

原因不明の障害に、各分野の専門家たちが集められた。

バックエンド、DBA、セキュリティ、SRE…全員が「俺の仕事は完璧だ」と主張する中、主人公「俺」は複雑に絡み合った障害の糸を解き明かしていく。

これは、専門性が生んだ悲劇であり、喜劇である。

シーン0：祝福と呪いのリリース

巨大モニターに映し出された新フロントエンド「The・World」。華々しいUI、滑らかなアニメーション。

PM民：「皆さん、やりました！『The・World』本日、無事リリースです！これも各分野のプロフェッショナルが集い、最高の仕事をした結果です！」

拍手喝采。各村の代表たちが誇らしげに胸を張る。

SRE「SLO？余裕で達成。Four Keysもオールグリーンだ」

セキュリティ「XSSもCSRFも脆弱性はゼロ。鉄壁だ」

バックエンド「APIレスポンスは50msを切った。俺の芸術的なコードのおかげだ」

俺：「（…なんか、フラグが乱立してる気がする…）」

シーン1：世界、沈黙

その瞬間、モニターが「ﾌﾟﾂﾝ」という音と共に真っ白になる。

🚨 画面真っ白 🚨

全社員のスマホからも「The・World」が消える。静まり返るリリース会場。誰かのシャンパングラスが床に落ちて割れる音が響く。

PM民：「…え？だ、誰だ！？誰がやった！？」

アラートが一斉に鳴り響き、SRE民の顔が青ざめる。

「SLIが崖から飛び降りていく！総員、第一種戦闘配置！」

俺：「（だから言わんこっちゃない…）」

シーン2：犯人は誰だ？（全員容疑者）

緊急対策会議室。巨大テーブルを囲む各村の代表たち。全員が疑心暗鬼。

PM「まず、最後にデプロイしたのは誰だ！SRE！」

SRE「待ってくれ！今回のデプロイはBlue/Greenだ。旧環境は無傷のはず。しかも切り替えは完璧だった！これはインフラかバックエンドの問題だ！」

バックエンド「はあ！？俺のAPIは全テスト通過済みだ！Postmanのコレクションを見ろ！完璧なJSONを返してる！」

DBA「そうだそうだ！DBのクエリも爆速だぞ！EXPLAINの結果、ALL Extraだ！」

容疑者たちの主張

セキュリティ「待て。リリース直前に怪しい海外IPからのアクセスが急増していた。俺が導入した新しいWAFルールが何か検知したのかもしれん。だが、それはお前らのコードに脆弱性があったからだろ？」

クラウド「コスト最適化のために昨日、CDNのキャッシュ設定を少し弄ったが…まさかな。影響範囲は軽微なはず…」

QA「我々の自動テストは1万ケース、全てパスしています！これは、テストケースにない『何か』が起きたとしか…つまり、仕様外の何か、そう、PMのせ…」

PM「俺のせいだと！？ユーザーの要望に応えるのが俺の仕事だ！」

俺：「（…ダメだこりゃ。全員、自分が正しいと信じて疑ってない。典型的なサイロ化障害だ…）」

シーン3：俺のターン！ログはすべてを語る

俺はノートPCを開き、各システムのログを鬼のような速さで突き合わせ始める。

// フロントエンドのエラーログ TypeError: undefined is not a function at UserProfile.render (main.js:1234)

俺：「まず、フロントエンドのエラーログ。『undefined is not a function』…想定外のデータ構造か」

バックエンド「ありえない！API仕様書通りだ！」

俺：「…いや、違うな。お前、パフォーマンス改善のためにネスト構造を一段浅くしてないか？『user.profile.name』が『user.name』になってるぞ」

バックエンド「あ…良かれと思って…。事前に『効率化した』ってSlackには書いたぞ！」

俺：「誰も見てない分報チャンネルで言うな！」

DBAの「最適化」

俺：「次、一部のユーザーから『データが途中で切れてる』という報告。DBA、お前だな？」

DBA「まさか！俺はむしろDBを高速化したが？」

俺：「プロフィール紹介文、以前はTEXT型だったのをVARCHAR(255)に変えただろ。自己紹介が長いユーザーのデータが見事にちぎれてる。結果、JSONが壊れてフロントが死んだ一因だ」

DBA「ぐっ…ストレージと速度のために…。まさか255文字以上書くやつがいるとは…」

⚠️ 原因その2：

VARCHAR(255)への変更により、長文プロフィールがトランケート

→ 不完全なJSONレスポンス → パースエラー

セキュリティの「念には念を」

俺：「そしてセキュリティ番、お前のWAFルール。『SQLインジェクションの疑い』でブロックログが大量に出てる。ブロックされたリクエストを見てみろ。…ユーザー名に『SELECT』って単語が入ってるだけじゃないか！」

セキュリティ「うっ…念には念を…まさか『DJ SELECT』なんて名前のユーザーがいるとは…性善説は脆弱性だとあれほど…」

⚠️ 原因その3：

過度に厳格なWAFルール

→ 正当なユーザー名（「SELECT」を含む）をブロック

→ ログインできないユーザーが多数発生

SREの「完璧な」デプロイ

俺：「SRE、お前のBlue/Greenデプロイ。完璧だと言ったな。だが、旧バージョンのJSをキャッシュしてるユーザーが新APIを叩くとどうなるか考えたか？DNSの切り替えとキャッシュのTTL（Time To Live）が喧嘩してるぞ」

SRE「そ、それは…エッジケースだ！ユーザーにスーパーリロードさせれば…」

俺：「できるか！画面真っ白なんだぞ！」

⚠️ 原因その4：

キャッシュされた旧JSが新APIを呼び出し

→ データ構造の不一致

→ 真っ白画面

クラウド担当の「コスト最適化」

俺：「クラウド担当、お前のCDN設定変更。これで一部リージョンからのアセット読み込みがS3直アクセスになって、表示が致命的に遅延。タイムアウトで死んでるユーザーが多発してる」

クラウド「月末の請求額が怖くて…経費削減は正義…」

⚠️ 原因その5：

CDN設定の変更により一部リージョンでS3直アクセス

→ レイテンシ増加

→ タイムアウト多発

PMの「リリース3時間前の変更」

俺：「とどめだ、PM！お前、リリース3時間前に『やっぱりユーザーアイコンは丸じゃなくて四角で！角丸は多めで！』って言っただろ！」

PM「ユーザー調査の結果、四角の方がエンゲージメントが高いと…」

俺：「そのせいで俺が書いたCSSの最終調整が吹っ飛んで、レイアウトが全部崩れてるんだよ！色々重なって、もうブラウザがレンダリングを諦めてる状態だ！」

⚠️ 原因その6：

直前のデザイン変更

→ CSSレイアウト崩壊

→ すべてが重なり合い、カオスへ

シーン4：そして誰も悪くなくなった

会議室は沈黙に包まれる。全員が犯人。しかし、全員が自分の職務に忠実だっただけ。

バックエンド「俺は…速度を上げたかった…」

DBA「俺は…DBを守りたかった…」

セキュリティ「俺は…会社を守りたかった…」

SRE「俺は…安定稼働を…」

全員がうなだれる。

俺：「そうさ。誰も悪くない。…強いて言うなら、コミュニケーションを取らなかった俺たち全員が悪いんだ」

そこからが本当のチームだった。

PM「もう『気合』で乗り切るのはやめます。これからは『対話』をリソースにします」

「隣の村のAPI仕様書を読め」

「デプロイ前には叫べ」

「『良かれと思って』はだいたい事故る」

フロントエンドエンジニアの役割

だからこそ、俺たちフロントエンドエンジニアは、誰よりも『翻訳者』でなきゃいけないのかもしれないな」

外伝：The・World村

十の事件簿

― 毎週どこかが壊れるけど、だいたい皆で笑って帰る ―

外伝について

第六章の大障害から数週間。

The・World村では、なぜか「毎週なにかが壊れる」という奇妙な風習（？）が生まれていた。

しかしそのたびに、俺たちは切り分けと特定を通じて、ちょっとずつ強くなっていく――。

これから、その10の事件を振り返る。

第1話「ログインボタンが押せない夜」

リリース当日、ユーザーからの悲鳴。「ログインボタンが押せません！」

フロント「俺のReactボタンは完璧だ！onClickもちゃんと生えてる！」

バックエンド「ログインAPIは200を返し続けている。呼ばれてさえいない！」

デザイン「ボタンは映えてる。UI的には世界を救えるレベル」

俺：「じゃあ、ブラウザのDevToolsでイベントを確認するか…」

俺：「…おいデザイン村。お前の『おしゃれな半透明ヘッダーバナー』、z-index9999で全画面にかぶさってるぞ。ボタンの上に透明な壁を置くな」

デザイン「えっ、だって『ヒーローセクションを画面いっぱいに』ってPMが…」

PM「（目をそらす）」

第1話：原因解析と学び

⚠️ 原因：透明なオーバーレイ要素（z-index: 9999）がボタンを物理的にふさいでいた

💥 結果：ログインボタンが一切クリックできず、ユーザーが村の門前で立ち尽くす

切り分けポイント

✓ イベントが発火しているかをDevToolsで確認

✓ 要素の上に何か重なっていないかをElements検証

✓ z-indexの値を全要素で確認

教訓：「見えないものが見えているものを壊す」- z-index事故は、透明な要素が原因のことが多い。「クリックできない」と言われたら、まず重なりを疑え。

第2話「消えた昨日の注文」

ある朝、サポート村から叫び声。「昨日の注文履歴だけ、全部消えてるんですけど！？」

DBA「DELETEなんてしてない！誓って！」

バックエンド「日付で検索してるだけだぞ？ WHERE created_at >= 今日 …あっ」

俺：「…それ、サーバーのタイムゾーンUTCで動いてるから、日本時間の『昨日』がごっそりフィルタされてるやつだな」

SRE「ログ見ると、0時を境に件数がピタッと0になってる。典型的TZ事故だ」

PM「昨日のキャンペーン成果が…レポートが真っ白…」

第2話：原因解析と学び

⚠️ 原因：タイムゾーンを考慮しない日付フィルタ（UTCとローカル時間のすれ違い）

💥 結果：実データは生きてるのに、UI上は「昨日だけ注文ゼロ」に見える怪現象

切り分けポイント

✓ DBの生データを直接SQLで確認（件数は本当にゼロ？）

✓ アプリケーション側の時刻変換ロジックを確認

✓ サーバーのタイムゾーン設定を確認（UTC? JST?）

✓ 日付境界付近でのデータ取得をテスト

教訓：「時間は世界共通ではない」- 日時を扱う時は常にタイムゾーンを意識せよ。DBはUTC、ユーザーはローカル時間で考える。

第3話「キャッシュに閉じ込められたバナー」

新キャンペーン開始。しかしユーザー画面には、なぜか2週間前に終わったセールバナーがドヤ顔で居座っている。

マーケ「新バナーに切り替えたのに！ABテストも仕込んだのに！」

クラウド「CDNのキャッシュ？まぁ…TTLは1週間にしてたけど…」

SRE「しかもバージョン付きURLつけてないから、全世界のエッジに古いやつが死ぬほど残ってるな」

俺：「フロント村でアセットに ?v=2 を付ける。CDN側でパスごとのキャッシュ削除を走らせよう」

クラウド「…ごめん、経費削減のために『キャッシュは長ければ長いほど偉い』って思ってた」

第3話：原因解析と学び

⚠️ 原因：強烈すぎるCDNキャッシュ＋バージョニングなし

💥 結果：終わったはずのセールが画面上だけ延長営業。ユーザーが混乱

切り分けポイント

✓ HTTPレスポンスヘッダ（Cache-Control, Expires）を確認

✓ CDNのキャッシュヒット/ミス状態を確認

✓ オリジンサーバーで正しいコンテンツが返っているか確認

✓ 異なるリージョンから確認（地理的なキャッシュ差分）

教訓：「更新したはずなのに反映されない」はキャッシュを疑え。静的アセットには必ずバージョン番号やハッシュ値をつけ、キャッシュバスティングを実装せよ。

第4話「フィーチャーフラグ・ロシアンルーレット」

新機能「ダークモード」を限定ロールだけ有効にするはずだった日。一部ユーザーだけ、謎の真っ黒画面に。

PM「フィーチャーフラグのロール設定は完璧だ。『社員だけ有効』にした」

セキュリティ「そのロール情報、クライアント側のLocalStorageを信じてるのか…？」

俺：「さらに悪いニュースだ。フロント村、環境変数名を FEATURE_DARKMODE と FEATURE_DARK_MODE で2種類やってる。片方だけ真に…」

SRE「つまり、社員ロールでもないごく一部のユーザーが、バグったダークモードを踏んでるわけか」

第4話：原因解析と学び

⚠️ 原因：フラグの読み取りミス（環境変数の綴り違い）＋権限情報をフロントだけで判断

💥 結果：特定ユーザーだけ謎の実験台にされる地獄

切り分けポイント

✓ 誰にどの条件でフラグが有効になるかをドキュメント化

✓ 環境変数名の一貫性を確認（typo、ケース違いに注意）

✓ フラグ判定のログを詳細に取得

✓ フロント・バックエンド両方でロール判定

教訓：「フィーチャーフラグは予測可能であれ」- 誰がどの機能を見られるか、明確な仕様とログで追跡可能にせよ。環境変数の命名規則は統一せよ。

第5話「翻訳キーのささやき」

多言語対応リリースの日。スペイン語ユーザーだけ、画面の半分が i18n.key.not.found で埋め尽くされる。

QA「テスト環境では全部英語で通したので…」

DevRel「コミュニティ翻訳を取り込んだけど、キーの一覧？そんなの見ずにノリで訳しました！」

俺：「本番だけ es-ES を有効化してるから、ステージングでは誰も気付かなかったパターンだな」

俺：「ログ見ると、missing translation が大量に。Fallbackも効いてない。翻訳キーのカバレッジをCIでチェックする仕組み、入れよう」

第5話：原因解析と学び

⚠️ 原因：本番だけ有効なロケールで、翻訳キーが半分足りていなかった

💥 結果：画面がシステム語と謎キーの羅列でホラー小説化

切り分けポイント

✓ 本番と同じロケール設定でステージング環境をテスト

✓ 翻訳キーの網羅性を静的解析でチェック（CI導入）

✓ 翻訳ファイルとコード内のキー使用箇所を突合

✓ Fallback言語が正しく動作するか確認

教訓：「多言語対応は全言語でテストせよ」- 翻訳キーの抜け漏れは静的解析で事前検出。本番だけ有効な設定は危険信号。

第6話「モバイルだけ落ちる呪い」

リリース後、古いAndroidユーザーからだけ悲鳴。「アプリが開いた瞬間に落ちる！」

フロント「ブラウザでは動いてる！PCのChromeだと完璧！」

モバイル「いや、Chrome 50 は optional chaining なんて知らないんだよ…」

俺：「…ポリフィルとトランスパイルのターゲット設定、ちゃんと見直そうな」

SRE「モバイルだけエラー率が跳ね上がっていたが、監視ダッシュボードは『全体平均』しか見てなかったから気付くの遅れたな」

第6話：原因解析と学び

⚠️ 原因：古いモバイルブラウザ非対応のモダン構文を、そのまま本番に出した

💥 結果：一部ユーザーだけ地獄。レビュー欄が火の海

切り分けポイント

✓ ユーザーセグメント別（OS、ブラウザ、バージョン）にエラー率を確認

✓ ビルドターゲット設定（.browserslistrc, tsconfig.json）を確認

✓ トランスパイル後のコードに古いブラウザで使えない構文がないか確認

✓ 実機またはBrowserStackで古いデバイスでテスト

教訓：「開発環境で動く ≠ 全ユーザーで動く」- サポートブラウザを明文化し、ビルド設定とポリフィルで互換性を担保せよ。監視は全体平均ではなくセグメント別に。

第7話「サードパーティ神の気まぐれ」

ある金曜の夜、決済画面だけが突然真っ白に。

バックエンド「うちのサーバーは正常！ログがそもそも来てない！」

SRE「外形監視も200を返してるが…」

俺：「決済ボタン押した瞬間、謎の外部JSを読みに行って、そこで止まってるな」

セキュリティ「決済プロバイダのステータスページ見た？世界的障害って書いてあるぞ」

PM「なんでウチの障害レポートに『他社のメンテ』が混ざるんだ…」

俺：「依存してるんだから、そこも含めてシステムだよ」

第7話：原因解析と学び

⚠️ 原因：サードパーティ決済SDKの大規模障害

💥 結果：決済フローが全国的に停止。The・World村も巻き添え

切り分けポイント

✓ 自前のサーバー・APIは正常か

✓ 外部サービスのステータスページを確認

✓ ネットワークタイムラインで外部リクエストの状態を確認

✓ 外部JSのロードエラー・タイムアウトをチェック

教訓：「外部依存は自分の責任範囲」- サードパーティサービスの障害も自システムの障害。ステータスページを監視し、タイムアウト処理とフォールバックを実装せよ。

第8話「レートリミットの逆襲」

新機能「まとめ通知」を出した翌日。一部ヘビーユーザーだけ、通知画面が「エラー祭り」に。

バックエンド「APIはちゃんと返してる…いや、ステータス429が増えてるな」

SRE「ゲートウェイで『1分間に10リクエストまで』って制限かけたの、誰？」

セキュリティ「ブルートフォース対策としては正しいと思ったんだが…」

俺：「フロント側でNリクエスト並列に飛ばしてるから、1画面開くだけで一瞬で制限超えてる。バッチ化しよう」

第8話：原因解析と学び

⚠️ 原因：APIゲートウェイのレートリミット設定と、フロントの雑な多重リクエスト設計の合わせ技

💥 結果：ヘビーユーザーだけが理不尽に締め出される

切り分けポイント

✓ HTTPステータス429（Too Many Requests）を確認

✓ APIゲートウェイのレートリミット設定値を確認

✓ フロント側で1画面あたり何リクエスト飛ばしているか計測

✓ バッチAPI化、リクエスト集約の可能性を検討

教訓：「制限値は実使用パターンと一緒に設計せよ」- レートリミットは必要だが、正常な使い方まで制限してはいけない。フロント側のリクエスト数を把握し、適切なバッチ化を。

第9話「CSSの一行で世界が崩れた日」

デザイン改善の小さなPR。変更内容はたった1行。

* { box-sizing: border-box; } → body * { box-sizing: border-box; }

デザイン「グローバル指定は良くないって聞いたから、スコープを絞っただけ…のはずだった」

俺：「その瞬間、モーダルの位置がずれて、SP版だけスクロール不能になったな」

QA「PCだけ見て『OKです！』って言いました！」

俺：「スタイルの差分をfeatureブランチで環境ごとに比較しよう。Visual Regression Testも入れたい」

第9話：原因解析と学び

⚠️ 原因：一見小さなCSS変更が、特定レイアウトと組み合わさって大崩壊

💥 結果：スマホユーザーがモーダルから出られなくなり、永遠にアンケート地獄

切り分けポイント

✓ CSS変更の影響範囲を複数の画面サイズで確認

✓ モーダル、ドロップダウンなど状態変化する要素で確認

✓ Visual Regression Testでスクリーンショット差分検出

✓ CSSセレクタの詳細度と継承を理解する

教訓：「CSSは予想外に伝播する」- グローバルなスタイル変更は全画面・全状態でテストせよ。Visual Regression Testで自動検出を。

第10話「移行スクリプトと失われたアイコン」

新システムへのユーザーデータ移行の日。移行後、半数以上のユーザーアイコンが、なぜか「謎の影絵」になっていた。

DBA「スクリプトは正常終了！エラー0件！」

バックエンド「古いストレージのパスを、新しいCDNのURLに置換しただけだぞ？」

クラウド「でも、そのバケットもう削除予約に入ってるんだよなぁ…」

俺：「ログを見ると、404を踏んだユーザーだけデフォルトアイコンの影絵が表示されてる。先にバケット削除スケジュールを組んで、その後ゆっくりDNS切り替えたから、移行中に消えた画像があるな」

PM「ユーザーから『黒いシルエットにされた』ってクレームが…」

第10話：原因解析と学び

⚠️ 原因：データ移行とストレージ削除の順番・タイミング設計ミス

💥 結果：一部ユーザーのアイコンが失われ、デフォルト影絵まみれに

切り分けポイント

✓ 移行計画をタイムライン付きで詳細に記述

✓ 旧システムの削除タイミングは移行完了後十分な期間を置く

✓ バックアップの存在とリストア手順を事前確認

✓ ロールバック手順を用意し、事前にテスト

教訓：「移行作業は逆順で考えよ」- 削除は最後、移行は段階的に。バックアップとロールバック手順なしに本番作業をするな。

エピローグ：毎週壊れるけど、毎週ちょっとだけ賢くなる村

俺：「気付けば、ここ数週間で10回もトラブルを踏んだわけだが…」

SRE「エラー率グラフは時々跳ねるけど、復旧時間（MTTR）は確実に短くなってる」

セキュリティ「ログの取り方も、だいぶ洗練されてきたな」

PM「障害報告会が、だんだんコメディショーになってきてない？」

俺：「いいんだよ。笑えるうちは、まだやり直せる。

大事なのは『誰を責めるか』じゃなくて、『どう切り分けて、どう直して、どう学んだか』だ」

今回のふりかえり

十の小さなトラブルは、どれも「良かれと思って」と「気づかなかった」が少しずつ積み重なって爆発した結果だった。

ひとつひとつを「誰のミスか」ではなく「どんな前提とコミュニケーションのズレがあったか」として分解すると、同じ村の中でも、隣の村の考え方や怖がっているポイントが、だんだん見えてくる。

ログを一緒に眺めて、タイムラインを引いて、付箋で原因と結果をつないでいく作業は、気づけば「ダメ出し会」ではなく「物語づくり」の場になっていた。

十本の小さな事件簿は、十個のチェックリストであると同時に、十通りの「こう考えればよかった」が詰まった教科書だ。

学びのまとめ：フロントエンドの視点から

1. 障害は複合的に発生する

単一の原因ではなく、複数の「良かれと思って」が重なり合って障害になる

2. ログは宝の山

フロントのエラーログ、バックエンドのアクセスログ、DBの実行ログを横断的に見る力

3. コミュニケーションは技術

「隣の村」の変更を把握し、影響を予測する習慣

4. 切り分けの型を持つ

「再現性」「ユーザーセグメント」「タイムライン」「環境差分」を軸に切り分ける

障害切り分けチェックリスト

✅ 再現性

・100%再現するか、特定条件下でのみ発生するか

・再現手順は何ステップか

✅ 影響範囲

・全ユーザーか、特定ユーザーか

・全機能か、特定機能か

・全環境か、本番だけか

✅ タイムライン

・いつから発生したか

・その前後に何か変更があったか

✅ ログと監視

・フロント、バックエンド、インフラの各層でエラーは出ているか

・外部サービスのステータスは正常か
